# Data Types in Music Recommendation Systems

> Derived from `music_recommender_research.md`

Recommendation systems run on two broad categories of data: **behavioral data** (what users do) and **content data** (what songs are). Each maps to a different filtering approach.

## Category 1: Behavioral Data (feeds Collaborative Filtering)

These are signals generated by user actions. They are implicit (passively recorded) or explicit (directly expressed).

### Implicit Signals — recorded automatically, no user effort

| Data Type | What it captures | Signal strength |
|---|---|---|
| **Play / Stream** | User started a song | Weak positive (could be accidental) |
| **Listen duration** | How much of the song was played (e.g., 80% vs. 10%) | Strong — short listens imply dislike |
| **Skip** | User skipped the track, and when | Negative signal; early skip = strong dislike |
| **Repeat play** | Song played again in the same session | Strong positive |
| **Search** | User searched for an artist/song by name | Strong interest signal |
| **Playlist add** | Song added to a personal playlist | Strong positive |
| **Share** | Song shared with another user | Very strong positive |
| **Autoplay continuation** | User let autoplay continue without skipping | Mild positive |
| **Session context** | Time of day, device type, location (home/commute) | Context modifier |

### Explicit Signals — the user consciously rated or acted

| Data Type | What it captures | Signal strength |
|---|---|---|
| **Like / Thumbs up** | User explicitly approved a song | Strong positive |
| **Dislike / Thumbs down** | User explicitly rejected a song | Strong negative — also suppresses similar tracks |
| **Save to library** | Added to personal collection | Strong positive |
| **Follow (artist)** | Long-term preference for an artist | Persistent positive |
| **Onboarding genre picks** | Genres chosen during account setup | Seed data for cold start |

## Category 2: Content Data (feeds Content-Based Filtering)

These are properties of the song itself, independent of any user behavior.

### Audio Features — measurable from the audio signal

| Data Type | Range / Format | What it means |
|---|---|---|
| **Tempo (BPM)** | ~40–220 BPM (float) | Speed of the track — slow ballad vs. fast dance track |
| **Energy** | 0.0 – 1.0 | Intensity and activity level; high-energy = loud, fast, noisy |
| **Valence** | 0.0 – 1.0 | Emotional positivity — high = happy/euphoric, low = sad/dark |
| **Danceability** | 0.0 – 1.0 | Suitability for dancing; based on rhythm regularity and beat strength |
| **Acousticness** | 0.0 – 1.0 | Confidence the track is acoustic (not electronic) |
| **Instrumentalness** | 0.0 – 1.0 | Likelihood of no vocals; >0.5 = likely instrumental |
| **Loudness** | ~−60 to 0 dB | Average overall loudness |
| **Key** | 0–11 (integer, pitch class) | Musical key (0 = C, 1 = C#, ..., 11 = B) |
| **Mode** | 0 or 1 | 0 = minor (darker), 1 = major (brighter) |
| **Time signature** | Integer (3, 4, 5, 7...) | Beats per bar — 4/4 is most common in pop |
| **Speechiness** | 0.0 – 1.0 | Presence of spoken words; high = podcast/rap, low = music |
| **Liveness** | 0.0 – 1.0 | Likelihood of a live audience being present |
| **Duration** | Milliseconds (int) | Length of the track |

### Deep Audio Features — extracted by neural networks

| Data Type | Format | What it means |
|---|---|---|
| **Audio embedding** | High-dimensional float vector | Latent representation of timbre, texture, and structure learned from raw spectrogram — captures things BPM and energy can't |
| **Spectrogram** | 2D array (frequency × time) | Visual representation of audio used as input to CNN models |

### Metadata — human-assigned or catalog data

| Data Type | Format | What it means |
|---|---|---|
| **Genre tags** | String / list of strings | Human-labeled or ML-inferred genre (e.g., "indie rock", "lo-fi hip hop") |
| **Mood labels** | String / list | Editorial tags like "chill", "focus", "workout", "heartbreak" |
| **Artist** | String / ID | Links to artist catalog, related artists |
| **Album** | String / ID | Groups tracks, implies era and artistic intent |
| **Release year** | Integer | Era/decade of the track — influences stylistic expectations |
| **Language** | String | Spoken/sung language of lyrics |
| **Lyrics text** | String | Full lyric text; used by NLP models to infer topic and mood |
| **Playlist name / description** | String | Editorial context — "Evening Jazz" tells a model about use case |

## Category 3: Contextual Data (modifies both systems)

Context data doesn't describe the user's long-term taste or a song's properties — it describes the *situation* of listening and adjusts recommendations in real time.

| Data Type | Examples | Effect on recommendations |
|---|---|---|
| **Time of day** | Morning, afternoon, late night | Morning → upbeat/energetic; late night → mellow |
| **Day of week** | Weekday vs. weekend | Weekend → party/social playlists |
| **Device** | Phone, smart speaker, desktop | Smart speaker → ambient/background; phone → active/commute |
| **Activity** | Workout, study, commute, dinner | Explicit context shifts mood and tempo targets |
| **Location** | Home, gym, transit | Inferred from GPS or manually set |
| **Listening session length** | Short burst vs. long session | Longer sessions → more variety introduced over time |
| **Current playlist / seed** | Song or playlist used to start a radio | Anchor point for CBF-based stream extension |

## How Data Types Map to Each System

```
BEHAVIORAL DATA          CONTENT DATA           CONTEXTUAL DATA
(likes, skips,           (tempo, energy,        (time, device,
 playlists, plays)        mood, genre)            activity)
      |                        |                       |
      v                        v                       v
Collaborative            Content-Based          Context-Aware
Filtering                Filtering              Adjustment Layer
      |                        |                       |
      └──────────── Hybrid Ranking Model ──────────────┘
                               |
                      Final Recommendation
```
